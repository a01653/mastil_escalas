MÁSTIL ESCALAS — INSTRUCCIONES DE DESARROLLO
=============================================

INSTALAR DEPENDENCIAS
---------------------
  npm install


EJECUTAR EN DESARROLLO (hot reload)
------------------------------------
  npm run dev


EJECUTAR PREVIEW (build optimizado local)
-----------------------------------------
  npm run build
  npm run preview


TESTS
-----

Todos los tests de una vez:
  npm test

Un fichero concreto:
  npx vitest run src/music/goldenCases.test.js
  npx vitest run src/music/chordDetectionEngine.test.js
  npx vitest run src/music/detectionInvariants.test.js
  npx vitest run src/music/mastilDebug.test.js

Qué significa cada fichero de tests:
  goldenCases.test.js          – Casos musicales concretos que no deben cambiar nunca.
                                 Si falla uno aquí, hay una regresión real en el motor.
  chordDetectionEngine.test.js – Tests directos del motor: fórmulas, candidatos, naming.
  detectionInvariants.test.js  – Invariantes generales (todo acorde min tiene b3, etc.).
  mastilDebug.test.js          – Tests del parser de digitaciones de consola (analyzeFrets).

Tests E2E principales (Playwright):
  npm run test:e2e

Tests E2E lentos / matriz de acordes:
  npm run test:e2e:chord-matrix

  Cuándo usarlos:
    - test:e2e: siempre que cambie UI, flujo de Acordes, copiar lecturas, modo estudio
      o cualquier interacción visible.
    - test:e2e:chord-matrix: obligatorio si se toca Acordes, voicings, inversiones,
      omit, extensiones, chips, naming o el flujo Investigar en mástil → Copiar en Acorde.


AUDITORÍA MASIVA DE DETECCIÓN DE ACORDES
-----------------------------------------
  npm run audit:chords

  Modo estándar (con cache por PC-set):
    npm run audit:chords

  Modo sin cache (voicings físicos individuales, más lento):
    npm run audit:chords -- --no-cache

  Qué hace: genera decenas de miles de voicings reales de guitarra (LowE→HighE,
  trastes 0-12), detecta acordes sobre cada uno y valida invariantes:
    - minor/dim/hdim no tienen b3 en missingLabels
    - dim/hdim siempre muestran b5
    - "Xm7b5" puro siempre tiene b3, b5, b7 visibles
    - slash /X coincide con bassPc de la lectura
    - no hay lecturas duplicadas con mismo contenido


AUDITORÍA MASIVA DE ACORDES / UI
---------------------------------
  npm run audit:chord-ui-matrix

  Qué hace: recorre una matriz amplia de combinaciones de la pestaña Acordes
  y valida coherencia entre:
    - fórmula solicitada
    - título del acorde
    - chips visibles
    - extensiones activas
    - omit/no1/no3/no5
    - bajo real del voicing
    - selector de inversión
    - notas reales del patrón
    - número de voicings

  Detecta estados musicales engañosos o inconsistentes, por ejemplo:
    - FORMULA_VOICING_MISMATCH
    - OMIT_NOT_PRESERVED
    - TITLE_STATE_MISMATCH
    - INSUFFICIENT_NOTES_MESSAGE_MISMATCH
    - INVERSION_LABEL_MISMATCH
    - BASS_REAL_MISMATCH
    - FUNCTIONAL_LABEL_MISMATCH
    - ORDINAL_LABEL_MIXED_WITH_BASS_LABEL
    - CHECKBOX_CHIP_MISMATCH

  Resultado esperado antes de entregar cambios de Acordes:
    0 FAIL, 0 WARN

  Si aparece un WARN, no ocultarlo. Debe quedar justificado en el informe o
  convertirse en un FAIL/corrección si realmente es un bug.


AUDITORÍA DE COPIAR LECTURAS A ACORDES
---------------------------------------
  npm run audit:copy-readings

  Qué hace: valida el flujo:
    Investigar en mástil → Posibles acordes → Copiar en Acorde

  Comprueba:
    - lectura primaria y candidatos secundarios
    - estructura copiada
    - extensiones
    - omit/no1/no3/no5
    - bajo/slash
    - patrón físico copiado cuando aplica
    - bloqueo de lecturas no representables en la UI

  Resultado esperado:
    todos los casos PASS



AUDITORÍA DEL MODO ESTUDIO
---------------------------
  npm run audit:study

  Recorre 12 casos de regresión (acordes en C Mayor) y valida:
    - Compatibilidad acorde/escala (diatónico / no diatónico)
    - Función armónica: Imaj7, IIm7, V7, IVmaj7, VIm7, VIIm7(b5)
    - Clasificación como dominante (dom vs. maj/min/hdim)
    - Dominantes secundarios: V7/Am = E7, V7/Dm = A7
    - Backdoor dominant: bVII7/G = F7
    - Distinción entre función principal y tonicizaciones alternativas
      (ej. G7 → II-V principal es Dm7-G7→Cmaj7; Am7-D7→G7 solo como tonicización)
    - Upper structures: A mayor/G7 como upper structure del V7 de C
    - Sustitución tritonal: G7 ↔ Db7
    - Spelling con bemoles: Eb no D#, Ab no G#, Cb no B en Db7
    - Advertencias de no-diatónico (Gmaj7, E7, A7, F7, Db7)

  Muestra por caso: acorde, escala activa, diatónico/no diatónico,
  clasificación dominante, función esperada y errores encontrados.


ANALIZAR DIGITACIÓN DESDE TERMINAL
------------------------------------
  npm run analyze:frets -- x5555x
  npm run analyze:frets -- x32010
  npm run analyze:frets -- x-10-9-8-7-x

  Con acorde de referencia (re-ranking igual que "Priorizar acorde de referencia" en la app):
  npm run analyze:frets -- x5555x --ref D7
  npm run analyze:frets -- 1320xx --ref Fmaj7
  npm run analyze:frets -- x5555x --ref C

  Salida JSON (para scripts y comparaciones automáticas):
  npm run analyze:frets -- x5555x --json
  npm run analyze:frets -- x5555x --ref D7 --json
  npm run analyze:frets -- 1320xx --ref Fmaj7 --json

  Detalle completo de lecturas (útil con --ref):
  npm run analyze:frets -- x5555x --ref D7 --all

  Qué hace: toma una digitación en formato LowE→HighE (igual que mastilDebug.analyzeFrets),
  calcula las notas reales, determina el bajo y llama al motor de detección.
  Muestra: patrón, notas detectadas, bajo, referencia, primary y lecturas ordenadas.

  Con --ref: aplica el mismo re-ranking que la app usa con la referencia armónica.
  Solo reordena candidatos cuya raíz coincida con la referencia; no inventa notas.
  Muestra "← por referencia" junto al candidato promovido por la referencia.

  Con --json: imprime SOLO JSON válido en stdout (sin colores, sin texto extra).
  Útil para que Codex/Claude/scripts comparen resultados automáticamente.
  Los errores también salen como JSON: { "ok": false, "error": "...", "code": "..." }

  NOTA: npm run imprime cabeceras en stderr, pero en PowerShell ConvertFrom-Json
  puede fallar si npm mezcla algo en stdout. Usar --silent o invocar node directamente:

    PowerShell / pipe a ConvertFrom-Json:
      npm --silent run analyze:frets -- x5555x --json | ConvertFrom-Json
      npm --silent run analyze:frets -- x5555x --ref D7 --json | ConvertFrom-Json

    Alternativa (invoca node directamente, sin cabeceras npm):
      node scripts/analyzeFrets.mjs x5555x --json | ConvertFrom-Json
      node scripts/analyzeFrets.mjs x5555x --ref D7 --json | ConvertFrom-Json

  Con --all: muestra fórmula, rank, raíz y bajo de todas las lecturas.
  Solo cambia la salida cuando se usa con --ref (sin --ref ya se muestra detalle completo).
  No afecta a --json (que siempre incluye todos los campos).

  Formatos de referencia aceptados:
    C, Cmaj7, C7, Cm, Cm7, Cm7b5, Cm7(b5), Cdim, Cdim7, Csus4, C7sus4
    Con alteraciones: Bb7, F#maj7, Dbmaj7, Eb7

  Formato de digitación:
    - Un carácter por cuerda (0-9) → "x5555x"
    - O tokens separados por - / , / espacio → "x-10-9-8-7-x"
    - 'x' = cuerda silenciada. Orden: cuerda 6 (LowE) → cuerda 1 (HighE).


ANÁLISIS DE PROGRESIONES / TONALIDAD DESDE CONSOLA
----------------------------------------------------
  npm run analyze:progression -- "F# | Bm | A/E | D/F#"
  npm run analyze:progression -- "C F G Am"
  npm run analyze:progression -- "Am | F | C | G"

  Uso:
    Permite probar el mismo motor de análisis tonal usado en la tarjeta
    "Analizador de tonalidad / progresión" (pestaña Acordes cercanos) sin abrir la UI.
    Reutiliza src/music/keyAnalysisEngine.js sin duplicar lógica.

  Separadores aceptados:
    |  coma  salto-de-línea  espacios simples o múltiples

  Slash chords:
    A/E  D/F#  (el bajo se extrae pero el análisis usa solo el acorde principal)

  Salida JSON (sin colores, útil para scripts):
    npm run analyze:progression -- "F# | Bm | A/E | D/F#" --json
    node scripts/analyzeProgression.mjs "F# | Bm | A/E | D/F#" --json

  Ejemplos representativos:
    npm run analyze:progression -- "C | F | G | Am"          → C mayor / A menor
    npm run analyze:progression -- "D | G | A | Bm"          → D mayor / B menor
    npm run analyze:progression -- "F# | Bm | A/E | D/F#"    → B menor (F# → dominante de Bm)
    npm run analyze:progression -- "Am | F | C | G"          → A menor / C mayor
    npm run analyze:progression -- "A7 | Dm | G | C"         → C mayor (cadencia II-V-I)
    npm run analyze:progression -- "B7 | Em | C | D"         → G mayor
    npm run analyze:progression -- "C | E7 | Am | F | G"     → C mayor o A menor
    npm run analyze:progression -- "Am | G | F | E"          → A menor (E = dominante de Am)

  Notas importantes:
    - El análisis es probabilístico por puntuación, no diatónico estricto.
    - Las tonalidades diatónicas estrictas (ver Acordes cercanos en la UI) son distintas
      de la "tonalidad probable" calculada por este motor.
    - El motor reconoce dominantes funcionales: F# → Bm en B menor, E → Am en A menor.
    - Las progresiones de blues (A7 D7 A7 E7 D7 A7) pueden dar resultados discutibles;
      el motor no detecta modos ni escalas de blues — solo Mayor y Menor natural.
    - Las alternativas con menor puntuación se muestran siempre que estén a 1 punto del top.

BATERÍA RECOMENDADA PARA PROBAR analyze:progression
------------------------------------------------------
  Ejecutar el script en modo --json para verificar automáticamente:

  A) Regresión estricta (los resultados NO deben cambiar):

    Diatónicos puros:
      node scripts/analyzeProgression.mjs "G | Em | C | D" --json
        → G mayor 100 %, E menor 100 %, sin funcionales ni fuera de tonalidad

      node scripts/analyzeProgression.mjs "F#m | Bm | A/E | D/F#" --json
        → D mayor primero; D mayor, A mayor, F# menor, B menor al 100 %

    Dominante funcional en menor (armónico V):
      node scripts/analyzeProgression.mjs "F# | Bm | A | D" --json
        → B menor primero 92 %; F# → dominante de Bm

      node scripts/analyzeProgression.mjs "E | Am | G | C" --json
        → A menor primero 92 %; E → dominante de Am

      node scripts/analyzeProgression.mjs "B7 | Em | C | D" --json
        → E menor primero; B7 → dominante de Em; G mayor como alternativa

    Dominante secundario:
      node scripts/analyzeProgression.mjs "A7 | Dm | G | C" --json
        → C mayor primero; A7 → dominante secundario de Dm

      node scripts/analyzeProgression.mjs "C | E7 | Am | F | G" --json
        → A menor primero 93 %; E7 → dominante de Am; E7 no es diatónico estricto

      node scripts/analyzeProgression.mjs "Dm7 | G7 | Cmaj7 | A7 | Dm7 | G7" --json
        → C mayor >= 80 %; A7 → dominante secundario de Dm

    Slash chords:
      node scripts/analyzeProgression.mjs "D/F# | G | A | D" --json
        → D mayor 100 %; D/F#, G, A, D todos diatónicos

  B) Comportamiento documentado / limitaciones (NO deben romper):

    Modal — bVII prestado (sin lógica de modos; resultado aproximado):
      node scripts/analyzeProgression.mjs "C | Bb | F | C" --json
        → No rompe; F mayor o D menor esperado (limitación: no detecta C mixolidio)

      node scripts/analyzeProgression.mjs "D | C | G | D" --json
        → No rompe; G mayor o E menor esperado (limitación: no detecta D mixolidio)

    Blues (limitación: dom7 se trata como mayor en matching diatónico):
      node scripts/analyzeProgression.mjs "A7 | D7 | A7 | E7 | D7 | A7" --json
        → A mayor 100 % (resultado razonable como aproximación; no es bug)

      node scripts/analyzeProgression.mjs "E7 | A7 | B7 | A7" --json
        → No rompe; E mayor o similar esperado

    Intercambio modal:
      node scripts/analyzeProgression.mjs "C | Ab | Bb | C" --json
        → No rompe; resultado ambiguo (fuera del modelo Mayor/Menor natural)

  Todos estos casos están cubiertos por tests automáticos en:
    src/music/keyAnalysisEngine.test.js  (grupos A y B)
    scripts/analyzeProgression.test.js   (grupos A y B — CLI)


MEJORAS FUTURAS PENDIENTES EN EL MOTOR DE ANÁLISIS TONAL
----------------------------------------------------------

1. Resolución real de dominantes secundarios

   Actualmente el motor puede etiquetar un acorde como dominante secundario si
   teóricamente es V de otro acorde de la tonalidad. En una mejora futura conviene
   comprobar también si resuelve realmente hacia ese acorde.

   Ejemplo:
     Bm | F# | A | E | G | D

   La tonalidad probable B menor es correcta.
   F# → dominante de Bm es correcto.
   E → dominante secundario de A es discutible si no resuelve a A;
   en ese contexto puede ser mejor marcarlo como acorde no diatónico,
   color dórico o posible dominante sin resolución.

2. Detección de blues (I7 - IV7 - V7)

   El motor trata dom7 como tríada mayor para el matching diatónico,
   lo que hace que A7 | D7 | E7 puntúe al 100% en A mayor.
   En una mejora futura, la progresión I7-IV7-V7 debería detectarse como
   "A blues" o etiquetarse de forma diferenciada de "A mayor diatónico".

3. Modos (mixolidio, dórico, etc.)

   Progresiones como C | Bb | F | C (bVII prestado → C mixolidio) o
   D | C | G | D (D mixolidio) se aproximan actualmente a F mayor o G mayor
   porque el motor solo conoce Mayor y Menor natural.
   En una mejora futura se podría puntuar modos específicos.


GENERAR MANUAL DE USUARIO (PDF)
---------------------------------
  Requiere que el servidor preview esté corriendo:
    npm run build
    npm run preview

  Luego, en otra terminal:
    npm run generate:manual

  Salida:
    manual/mastil_escalas_manual.pdf   — manual completo (A4, ~10 págs.)
    manual/manual.html                  — fuente HTML del manual
    manual/*.png                        — capturas intermedias

  El script abre Chrome headless, navega automáticamente por todas las
  secciones (Escala, Patrones, Ruta, Acordes, Modo Estudio, Investigar,
  Acordes cercanos, Standards), toma capturas reales y genera el PDF.

  NOTA: la carpeta manual/ está en .gitignore (binarios derivados).
  Regenerar el manual siempre desde el source con este comando.

  Para especificar una URL manualmente:
    node scripts/generateManual.mjs --url http://localhost:4174/mastil_escalas/

AYUDA DE CONSOLA (window.mastilDebug)
--------------------------------------
Cuándo está disponible:

  1. Siempre activo con `npm run dev` (modo desarrollo local).

  2. En producción / GitHub Pages: añadir ?debug=1 a la URL:
       https://.../mastil_escalas/?debug=1
     Sin ese parámetro, window.mastilDebug no existe en producción.

  El chunk del módulo SÍ está incluido en el bundle de producción
  (para poder cargarlo bajo demanda), pero solo se carga si ?debug=1.
  En una sesión normal sin ?debug=1, el módulo no se ejecuta ni se registra.

  NOTA: `npm run preview` sirve el bundle de producción, así que también
  requiere ?debug=1 para activar el debug (http://localhost:4173/mastil_escalas/?debug=1).

Analizar notas por nombre:
  window.mastilDebug.analyzeNotes(["D", "G", "C", "E"], "D")
  window.mastilDebug.analyzeNotes(["C", "E", "G"])

Analizar digitación en formato LowE→HighE (6 cuerdas):
  window.mastilDebug.analyzeFrets("x5555x")    // acorde desde cejilla/trastes
  window.mastilDebug.analyzeFrets("x32010")    // C mayor abierto
  window.mastilDebug.analyzeFrets("x-10-9-8-7-x")  // trastes de 2 dígitos separados por -

  Formato: un carácter por cuerda (0-9) o separados por - / , / espacio.
           'x' = cuerda silenciada.
           Orden: string 6 (LowE) → string 1 (HighE).

Ambas funciones imprimen el diagnóstico en consola y devuelven el objeto resultado.
